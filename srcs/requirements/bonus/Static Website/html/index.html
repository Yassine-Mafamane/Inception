System Administration with Docker

Project Overview

The Inception project is a comprehensive system administration exercise focused on containerization and infrastructure setup using Docker technology. The goal is to create a small but complete infrastructure composed of multiple services, each running in dedicated containers.

Core Requirements

  • NGINX: Web server configured with TLSv1.2/TLSv1.3 for secure HTTPS connections
  • WordPress: CMS with php-fpm
  • MariaDB: Database server for WordPress
  • Redis: In-memory caching server to accelerate WordPress performance
  • FTP: File Transfer Protocol server for remote file management
  • Adminer: Web-based database administration tool for MariaDB
  • Volumes: Persistent storage for database and website files
  • Docker Network: Secure communication between containers
Key Constraint: NGINX container serves as the sole entry point to the infrastructure, accessible only via port 443 with TLS encryption.

Docker & Containerization

Docker is an open platform for developing, shipping, and running applications that fundamentally changes how we manage infrastructure. It enables you to package and run applications in lightweight, isolated environments called containers. Unlike traditional virtual machines that require a full operating system for each instance, containers share the host's kernel while maintaining isolation, making them significantly more efficient and faster to start. Each container includes everything needed to run the application‚Äîcode, runtime, libraries, and dependencies‚Äîensuring consistency across different environments without relying on what's installed on the host.

The power of Docker lies in its portability and consistency. Containers can run seamlessly on a developer's laptop, physical or virtual machines in data centers, cloud providers, or any mixture of environments. This portability enables reliable workflows: development teams write code in Docker environments, push containers for testing, and when bugs are discovered, they can quickly fix issues and redeploy the updated container with confidence that it will work identically everywhere. This approach, known as continuous delivery, eliminates the classic "it works on my machine" problem. By containerizing applications with all their dependencies, this project leverages Docker to create an isolated, reproducible, and scalable infrastructure that can be shared and deployed with guaranteed consistency.

Why Docker?

  • Isolation: Each service runs in its own container with dedicated resources
  • Portability: The entire infrastructure can be deployed anywhere
  • Efficiency: Containers share the host OS kernel, reducing overhead
  • Reproducibility: Infrastructure as code ensures consistent deployments

Implementation Details

All Docker images are built from scratch using custom Dockerfiles based on Alpine or Debian. The project uses Docker Compose to orchestrate multiple containers, defining services, networks, and volumes in a single configuration file. Each container is designed to run a single process as PID 1, following Docker best practices. A Makefile automates the entire deployment workflow, handling data directory creation, container orchestration, and cleanup operations for consistent and reproducible deployments.

Vault - Secret Management (Additional Service)

HashiCorp Vault was chosen as an additional bonus service to provide secure, centralized secret management for the infrastructure. This addresses a critical security concern in containerized environments.

Why Vault?

  • Security: Secrets are encrypted at rest and in transit
  • Centralized Management: Single source of truth for all credentials
  • Dynamic Secrets: Can generate credentials on-demand with automatic expiration
  • Audit Logging: Complete audit trail of secret access
  • Access Control: Fine-grained policies control who can access what

Implementation

Vault runs as a dedicated container in the infrastructure. Instead of hardcoding passwords in environment variables or files, sensitive credentials (database passwords, API keys, SSL certificates) are stored in Vault's encrypted storage.

Security Flow:
  1. Services authenticate with Vault using the AppRole method for automated credential retrieval
  2. Vault generates TLS certificates signed by a Vault-managed root CA, enabling certificate revocation, expiration checks, and centralized trust management
  3. Database credentials are dynamically generated with short lifespans and can be revoked instantly in case of breaches or team member departures
  4. Vault verifies identities, enforces access policies, and grants only the necessary secrets to each service
  5. All secrets are injected at runtime and never stored in container images or configuration files
  6. In case of security incidents or credential leaks, Vault can be sealed to prevent access until the situation is resolved

This approach eliminates the risk of accidentally committing secrets to Git, provides rotation capabilities for credentials, and offers a professional-grade solution for managing sensitive data in production environments.

Areas for Improvement

While this project demonstrates a functional containerized infrastructure, several security enhancements could be implemented in a production environment:

Security Hardening

  • Non-Root Container Users: Containers should run as non-privileged users instead of root to minimize the impact of potential container breakouts and follow the principle of least privilege.
  • Vault Root Token Management: The root token should be revoked after initial setup and authentication should rely solely on AppRole or other secure methods. Root tokens should never be left accessible in production environments.
  • Encrypted Communication:
    • Vault communication should use TLS/SSL encryption to protect secrets in transit
    • FTP should be replaced with SFTP or FTPS to encrypt file transfers
  • Secret Rotation: Implement automated secret rotation policies for long-lived credentials and certificates.
